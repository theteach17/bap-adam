# bap-adam — Activity Name Patch

แพ็กเกจนี้เป็นไฟล์พร้อมวางแทนที่ระบบ Google Apps Script Web App โครงการ `bap-adam` โดยพัฒนาต่อจากเวอร์ชันที่มี Security Session Patch แล้ว

## สรุปสิ่งที่ปรับปรุง

1. เพิ่มฟิลด์ `ชื่อกิจกรรม` ในแบบฟอร์ม `ขอเลขทะเบียนเอกสาร`
2. วางฟิลด์ `ชื่อกิจกรรม` ถัดจากฟิลด์ `ชื่อเอกสาร`
3. เมื่อกรอก `รหัสกิจกรรม` แล้วพบข้อมูลในชีต `LinkBAP` ระบบจะเติมข้อมูลอัตโนมัติ:
   - ชื่อกิจกรรม
   - กลุ่มบริหาร
   - กลุ่มงาน
   - ผู้รับผิดชอบ
   - Email
   - โครงการตามแผนปฏิบัติการ
4. รองรับการกด `Enter` ในช่อง `รหัสกิจกรรม` เพื่อค้นหาทันที
5. ถ้ากรอกรหัสกิจกรรมแล้วไม่พบข้อมูล ระบบจะแจ้งเตือนและล้างข้อมูลที่เกี่ยวข้อง
6. ถ้าไม่กรอกรหัสกิจกรรม ระบบยังอนุญาตให้กรอกข้อมูลเองได้ตาม logic เดิม แต่บังคับให้ `ชื่อกิจกรรม` ขึ้นต้นด้วยคำว่า `กิจกรรม`
7. ก่อนบันทึก ระบบจะแสดง SweetAlert สรุปข้อมูลให้ผู้ใช้ยืนยันก่อนสร้างเลขทะเบียน
8. บันทึก `ชื่อกิจกรรม` ลงชีต `ReportNo` คอลัมน์ L
9. ไม่แทรกคอลัมน์กลางตารางเดิม เพื่อไม่กระทบคอลัมน์ A-K และ logic เดิมของระบบ
10. ซ่อนคอลัมน์ `ชื่อกิจกรรม` จาก dashboard เพื่อไม่เปลี่ยน layout ตารางหลักเดิม

## ไฟล์ในแพ็กเกจ

- `รหัส.gs` — backend Google Apps Script ที่ปรับแล้ว
- `Index.html` — frontend หลักที่ปรับแล้ว
- `Login.html` — ไฟล์เดิม ไม่ได้เปลี่ยน
- `appsscript.json` — manifest เดิม ไม่ได้เพิ่ม scope
- `README_SECURITY_PATCH.md` — เอกสาร security patch เดิม
- `README.md` — เอกสารฉบับนี้

## โครงสร้างข้อมูลที่เพิ่ม

ต้องมีคอลัมน์ใหม่ในชีต `ReportNo`

| คอลัมน์ | Header |
|---|---|
| L | ชื่อกิจกรรม |

ระบบมี helper `ensureReportNoActivityNameColumn_()` ช่วยตรวจสอบคอลัมน์ L:
- ถ้า L1 ว่าง ระบบจะตั้งค่าเป็น `ชื่อกิจกรรม` ให้อัตโนมัติ
- ถ้า L1 มีค่าอื่นอยู่แล้ว ระบบจะหยุดบันทึกและแจ้ง error เพื่อป้องกันการเขียนทับข้อมูลเดิม

## เหตุผลที่ใช้คอลัมน์ L

ระบบเดิมใช้คอลัมน์ A-K ของ `ReportNo` อยู่แล้ว เช่น

- B = หมายเลขเอกสาร
- C = ชื่อเอกสาร
- D = กลุ่มบริหาร
- E = กลุ่มงาน
- F = ผู้รับผิดชอบ
- G = บันทึกข้อความ
- H = ลิงก์ไฟล์
- I = โครงการ
- J = อีเมล
- K = รหัสกิจกรรม

จึงเพิ่ม `ชื่อกิจกรรม` ที่คอลัมน์ L เพื่อไม่ให้ตำแหน่งคอลัมน์เดิมเปลี่ยน และไม่กระทบการส่งรายงานหรือการอัปโหลดไฟล์เดิม

## วิธีติดตั้ง

1. สำรอง Apps Script project เดิม
2. สำรอง Google Spreadsheet ฐานข้อมูล `Project Adam`
3. เปิด Apps Script Editor
4. วางไฟล์แทนที่ดังนี้:
   - นำเนื้อหา `รหัส.gs` ไปวางแทนไฟล์ backend เดิม
   - นำเนื้อหา `Index.html` ไปวางแทนไฟล์หน้าเว็บหลักเดิม
   - `Login.html` และ `appsscript.json` สามารถวางทับได้หรือคงไฟล์เดิมไว้ได้ เพราะไม่มีการแก้ไขสำคัญ
5. ตรวจสอบชีต `ReportNo`
   - ถ้าคอลัมน์ L ว่าง ให้ระบบสร้าง header ให้เองตอนบันทึกครั้งแรกได้
   - หรือใส่ `ชื่อกิจกรรม` ที่ L1 ด้วยตนเองก่อน deploy ก็ได้
6. กด Save
7. Deploy เป็น version ใหม่
8. ทดสอบผ่าน Test deployment ก่อนใช้งานจริง

## Validation ที่เพิ่ม

### กรณีกรอกรหัสกิจกรรมและพบข้อมูล

- เติมข้อมูลอัตโนมัติจาก `LinkBAP`
- ฟิลด์ `ชื่อกิจกรรม` เป็น read-only
- อนุญาตให้บันทึกหลังยืนยัน SweetAlert

### กรณีกรอกรหัสกิจกรรมแต่ไม่พบข้อมูล

- แจ้งเตือนว่าไม่พบรหัสกิจกรรม
- ล้างข้อมูล:
  - ชื่อกิจกรรม
  - กลุ่มบริหาร
  - กลุ่มงาน
  - ผู้รับผิดชอบ
  - Email
  - โครงการตามแผนปฏิบัติการ
- ไม่อนุญาตให้บันทึกจนกว่าจะแก้รหัสให้ถูกต้อง หรือเคลียร์รหัสกิจกรรมให้ว่าง

### กรณีไม่กรอกรหัสกิจกรรม

- อนุญาตให้กรอกข้อมูลเอง
- บังคับกรอก `ชื่อกิจกรรม`
- `ชื่อกิจกรรม` ต้องขึ้นต้นด้วยคำว่า `กิจกรรม`

### ก่อนบันทึก

- แสดง SweetAlert สรุปข้อมูลทั้งหมด
- ถ้าผู้ใช้กดยกเลิก จะไม่บันทึก
- ถ้าผู้ใช้ยืนยัน จึงเรียก `saveData()` และสร้างเลขทะเบียนตาม logic เดิม

## แผนทดสอบขั้นต่ำ

1. เปิดหน้า Login และเข้าสู่ระบบ
2. เปิดแบบฟอร์ม `ขอเลขทะเบียนเอกสาร`
3. กรอกรหัสกิจกรรมที่มีอยู่ แล้วกด Enter
   - ต้องเติมชื่อกิจกรรมและข้อมูลอื่นครบ
4. กรอกรหัสกิจกรรมที่มีอยู่ แล้วคลิกออกจากช่อง
   - ต้องเติมข้อมูลได้เหมือนเดิม
5. กรอกรหัสกิจกรรมผิด
   - ต้องแจ้งเตือนและล้างข้อมูลที่เกี่ยวข้อง
6. กรอกรหัสกิจกรรมผิดแล้วกดบันทึก
   - ต้องไม่บันทึก
7. ไม่กรอกรหัสกิจกรรม แต่กรอกชื่อกิจกรรมขึ้นต้นด้วย `กิจกรรม`
   - ต้องผ่าน validation
8. ไม่กรอกรหัสกิจกรรม และชื่อกิจกรรมไม่ขึ้นต้นด้วย `กิจกรรม`
   - ต้องไม่ผ่าน validation
9. กดบันทึกข้อมูล
   - ต้องแสดง SweetAlert สรุปข้อมูล
10. กดยกเลิกใน SweetAlert
   - ต้องไม่บันทึก
11. กดยืนยันใน SweetAlert
   - ต้องบันทึกข้อมูลและสร้างเลขทะเบียนตามเดิม
12. ตรวจชีต `ReportNo`
   - คอลัมน์ L ต้องมี `ชื่อกิจกรรม`
13. ตรวจ dashboard
   - layout ตารางหลักต้องไม่เปลี่ยน
14. ทดสอบส่งรายงานฉบับสมบูรณ์
   - ต้องยังทำงานตามเดิม
15. ทดสอบแจ้งกิจกรรมที่ไม่ได้ดำเนินการ
   - ต้องยังทำงานตามเดิม

## Known Google Apps Script limitations

- Apps Script มี execution limit ต่อครั้งประมาณ 6 นาที
- การอ่าน/เขียน Google Sheets จำนวนมากอาจช้าหากมีผู้ใช้พร้อมกันสูง
- Drive resumable upload ยังขึ้นกับ quota ของ UrlFetchApp และ Drive API
- การใช้งานพร้อมกันจำนวนมากควรหลีกเลี่ยงการบันทึกเอกสารพร้อมกันจำนวนมากในเวลาเดียวกัน
- ระบบนี้ยังใช้ username/password จากชีต `Credential` ตามโครงสร้างเดิม แม้มี session guard แล้ว แต่ยังไม่ใช่ password hash

## หมายเหตุด้านความปลอดภัยและผลกระทบ

- ไม่แก้ `appsscript.json`
- ไม่เพิ่ม OAuth scope
- ไม่เปลี่ยน deployment mode
- ไม่เปลี่ยน logic การสร้างเลขทะเบียน
- ไม่เปลี่ยนคอลัมน์ A-K ของ `ReportNo`
- ไม่เปลี่ยน workflow ส่งรายงานและแจ้งไม่ได้ดำเนินการ
- เพิ่ม server-side validation ใน `saveData()` เพื่อป้องกันการ bypass validation จากฝั่ง browser

## Rollback

หากต้องย้อนกลับ ให้ใช้ไฟล์สำรองก่อนติดตั้ง patch นี้ และตรวจสอบว่าคอลัมน์ L ใน `ReportNo` ไม่มีผลกระทบต่อสูตรเดิม หากต้องเก็บข้อมูลไว้สามารถคงคอลัมน์ L ได้ เพราะคอลัมน์นี้ไม่กระทบ A-K เดิม

## Report Name Validation Patch เพิ่มเติม

เพิ่ม validation สำหรับฟิลด์ `ชื่อเอกสาร` ในแบบฟอร์ม `ขอเลขทะเบียนเอกสาร` เพื่อบังคับให้ชื่อเอกสารขึ้นต้นด้วยรูปแบบที่กำหนดเท่านั้น

รูปแบบที่อนุญาต:

1. `รายงานผลการดำเนินกิจกรรม...`
2. `บันทึกข้อความชี้แจงไม่ดำเนินกิจกรรม...`
3. `เอกสาร...`

สิ่งที่แก้เพิ่มเติม:

- ปรับ placeholder และข้อความช่วยเหลือใต้ช่อง `ชื่อเอกสาร` ให้สอดคล้องกับกติกาใหม่
- เพิ่ม client-side validation ใน `Index.html` ก่อนแสดง SweetAlert ยืนยันการบันทึก
- เพิ่ม server-side validation ใน `saveData()` ของ `รหัส.gs` เพื่อป้องกันการ bypass จากฝั่ง browser
- ไม่เปลี่ยนโครงสร้างฐานข้อมูล ไม่เพิ่มคอลัมน์ใหม่ และไม่กระทบคอลัมน์ A-L ของ `ReportNo`
- ไม่เปลี่ยน logic การสร้างเลขทะเบียน การส่งรายงาน การอัปโหลดไฟล์ หรือระบบ session/auth เดิม

กรณีทดสอบเพิ่มเติม:

1. กรอก `รายงานผลการดำเนินกิจกรรมพัฒนาทักษะภาษาอังกฤษ` ต้องผ่าน
2. กรอก `บันทึกข้อความชี้แจงไม่ดำเนินกิจกรรมอบรมเชิงปฏิบัติการ` ต้องผ่าน
3. กรอก `เอกสารสรุปข้อมูลการดำเนินงาน` ต้องผ่าน
4. กรอก `รายงานผลการดำเนินงาน...` ต้องไม่ผ่าน
5. กรอก `สรุปกิจกรรม...` ต้องไม่ผ่าน
6. กรอก `บันทึกข้อความไม่สามารถดำเนินกิจกรรม...` ต้องไม่ผ่าน

## Report Submission Statistics Patch เพิ่มเติม

แพตช์นี้ปรับปรุงแบบฟอร์ม `ส่งรายงานฉบับสมบูรณ์` โดยพัฒนาต่อจากชุดที่มี Session/Auth Guard, Activity Name Patch และ Report Name Validation Patch แล้ว

### สิ่งที่เพิ่มในหน้า `ส่งรายงานฉบับสมบูรณ์`

1. เพิ่มฟิลด์ `ชื่อกิจกรรม` ในส่วน `ข้อมูลทั่วไป (Read Only)`
   - แสดงใต้ `ชื่อรายงาน`
   - แสดงแบบ full width เหมือน `โครงการตามแผนฯ`
   - ดึงจาก `ReportNo!L`
   - หากข้อมูลเก่าไม่มีค่าใน `ReportNo!L` แต่มี `รหัสกิจกรรม` ระบบจะ fallback ไปค้นหาจาก `LinkBAP`

2. เพิ่มฟิลด์ `บรรลุผลที่คาดว่าจะได้รับ`
   - เป็น dropdown มีค่า `บรรลุ` และ `ไม่บรรลุ`
   - บังคับกรอกทุกประเภทเอกสาร

3. เพิ่มฟิลด์ค่าสถิติ 4 ช่อง
   - `ค่า X̄ (X Bar) ความพึงพอใจ`
   - `ค่า SD ความพึงพอใจ`
   - `ค่า X̄ (X Bar) ของผลการบริหารกิจกรรม`
   - `ค่า SD ผลการบริหารกิจกรรม`

4. เพิ่ม SweetAlert สรุปข้อมูลก่อนอัปโหลดไฟล์และบันทึกจริง
   - ถ้ากดยกเลิก ระบบจะไม่อัปโหลด PDF และไม่บันทึกข้อมูล
   - ถ้ายืนยัน ระบบจึงเริ่มอัปโหลดและบันทึกตาม workflow เดิม

### เงื่อนไขการบังคับกรอกค่าสถิติ

ระบบจำแนกประเภทจาก `ชื่อรายงาน` เป็นหลัก:

- ถ้าขึ้นต้นด้วย `รายงานผลการดำเนินกิจกรรม`
  - บังคับกรอกค่าสถิติทั้ง 4 ช่อง
- ถ้าขึ้นต้นด้วย `บันทึกข้อความชี้แจงไม่ดำเนินกิจกรรม`
  - ไม่บังคับกรอกค่าสถิติ
- ถ้าขึ้นต้นด้วย `เอกสาร`
  - ไม่บังคับกรอกค่าสถิติ

### รองรับเอกสารเดิม `/2568`

ถ้าเลขเอกสารลงท้ายด้วย `/2568` และชื่อเอกสารไม่ตรงกับ prefix มาตรฐานใหม่ ระบบจะแสดง dropdown `ประเภทเอกสารสำหรับการตรวจค่าสถิติ` ให้ผู้ใช้เลือกเอง:

- `รายงานผลการดำเนินกิจกรรม` → บังคับค่าสถิติ
- `บันทึกข้อความชี้แจงไม่ดำเนินกิจกรรม` → ไม่บังคับค่าสถิติ
- `เอกสารอื่น ๆ` → ไม่บังคับค่าสถิติ

ค่าประเภทที่ใช้ตัดสินนี้จะถูกบันทึกลง `ReportSubmit!Y` เพื่อ audit ย้อนหลัง

### Validation ค่าสถิติ

ช่องค่า X̄ ทั้ง 2 ช่อง:

- ต้องเป็นเลขทศนิยม 2 หลักเท่านั้น เช่น `4.50`
- ต่ำสุด `0.01`
- สูงสุด `5.00`

ช่องค่า SD ทั้ง 2 ช่อง:

- ต้องเป็นเลขทศนิยม 2 หลักเท่านั้น เช่น `0.50`
- ต่ำสุด `0.01`
- สูงสุด `1.00`

ถ้าเอกสารไม่บังคับค่าสถิติ ผู้ใช้สามารถเว้นว่างได้ แต่ถ้ากรอกค่าเข้ามา ระบบจะตรวจรูปแบบและช่วงค่าตามเงื่อนไขเดียวกัน

### โครงสร้างข้อมูลใหม่ใน `ReportSubmit`

เพิ่มเฉพาะคอลัมน์ด้านขวาสุด เพื่อไม่กระทบคอลัมน์ A-S เดิม

| คอลัมน์ | Header |
|---|---|
| T | ชื่อกิจกรรม |
| U | บรรลุผลที่คาดว่าจะได้รับ |
| V | ค่า X̄ (X Bar) ความพึงพอใจ |
| W | ค่า SD ความพึงพอใจ |
| X | ค่า SD ผลการบริหารกิจกรรม |
| Y | ประเภทเอกสารสำหรับการตรวจค่าสถิติ |

หมายเหตุ: `ค่า X̄ (X Bar) ของผลการบริหารกิจกรรม` ยังคงบันทึกไว้ที่คอลัมน์ M เดิม เพื่อไม่กระทบข้อมูลและสูตรเดิม

ระบบมี helper `ensureReportSubmitExtendedColumns_()` เพื่อตรวจหัวคอลัมน์ T-Y:

- ถ้า T1:Y1 ว่าง ระบบจะตั้งหัวคอลัมน์ให้อัตโนมัติ
- ถ้ามีค่าอื่นอยู่แล้ว ระบบจะหยุดบันทึกและแจ้ง error เพื่อป้องกันการเขียนทับข้อมูลเดิม

### จุดที่ไม่เปลี่ยน

- ไม่แก้ `appsscript.json`
- ไม่เพิ่ม OAuth scope
- ไม่แก้หน้า Login
- ไม่แก้ Sidebar และ Dashboard หลัก
- ไม่แก้ workflow อัปโหลดไฟล์ PDF เดิม
- ไม่เปลี่ยนคอลัมน์ A-S ของ `ReportSubmit`
- ไม่ย้ายข้อมูลคอลัมน์ M เดิม
- ไม่บังคับ migrate ข้อมูลเก่า
